## Motivation

Workspace re-zeroing is the dominant per-call cost for `iris.ccl.all_reduce` on small/medium buffers and contributes a measurable share of the gap to RCCL on MI300X. The public wrapper at `iris/ccl/all_reduce.py` set `workspace.prepared = False` after every call, which forced `launch()` to re-run the full preamble (output `zero_()`, ring-buffer / flags / locks `zero_()`, and a `ctx.barrier()`) on every subsequent invocation — even when the next call had identical variant, shape, dtype, and world size.

This bug was confirmed in K-4574. The fix lets `prepared` persist across homogeneous calls and relies on the existing launch-side dirty-flag guard (`needs_prepare`) to re-run the preamble exactly when something material changes (variant / shape / dtype / num_rings / required buffers missing).

## Technical Details

- File: `iris/ccl/all_reduce.py`
- Removed the unconditional `workspace.prepared = False` reset that ran after every successful `launch(...)` (was at line 77 of the public wrapper).
- Added an inline note explaining the contract: the launch-side `needs_prepare` guard at `iris/ccl/triton/all_reduce.py:740` already re-runs `all_reduce_preamble` when:
  - `workspace is None`, or
  - `not workspace.prepared` (e.g. user/caller invalidates explicitly), or
  - `variant`, `shape`, or `dtype` changes, or
  - ring-specific params (`num_rings`, `flags_per_tile`) change, or
  - spinlock requires `locks` to be allocated.
- No changes to kernels, no changes to the preamble itself, no changes to the dirty-flag guard. Pure removal of the redundant invalidation.

## Test Plan

8 x MI300X (cluster c42, node f06u13), `rocm/pytorch:rocm7.2_ubuntu24.04_py3.12_pytorch_release_2.10.0`.

1. Functional regression — full `tests/ccl/test_all_reduce.py` suite at world_size=8:

2. Source / dirty-flag invariants (offline, in-container):
   ```
   python3 verify_patch.py
   ```
3. Per-call latency micro-benchmark, `8192 x 4096` bf16 (64 MiB), ws=8, 100 iters, three variants:
   ```
   torchrun --standalone --nproc-per-node=8 bench_K4640.py
   ```
   The benchmark forces the old behavior (`workspace.prepared = False` after each call) for the baseline measurement and uses the patched code path for the comparison.

Per-call benchmark (8192x4096 bf16, ws=8, 100 iters):

| variant   | baseline (us) | patched (us) | delta (us) | reduction |
|-----------|--------------:|-------------:|-----------:|----------:|
| two_shot  |        773.47 |       753.98 |      19.49 |    2.58 % |
| atomic    |     488813.32 |    425822.17 |   62991.16 |   14.79 % |
| one_shot  |       3499.36 |      3388.74 |     110.62 |    3.26 % |

The atomic variant shows the largest absolute saving because its preamble does `output_tensor.zero_()` plus a `ctx.barrier()` — both were being repeated on every call before this fix. For the timed variants, the patched code paths achieve a strictly higher effective bandwidth.
